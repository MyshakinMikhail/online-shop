# AGENTS.md

Инструкции для AI-агентов, которые работают с этим репозиторием. Файл должен помогать быстро понять проект, запускать правильные команды и не ломать чужие изменения.

## Проект

Fullstack интернет-магазин одежды.

- `frontend/` - React 19, TypeScript, Vite, Redux Toolkit, React Router, Ant Design, CSS Modules.
- `backend/` - Node.js, Express, TypeScript, Sequelize, PostgreSQL, Vitest, Supertest.
- API-контракты и бизнес-контекст лежат в `API_CONTRACTS.md`.
- Проектная документация лежит в `README.md`, но перед изменениями проверяй фактический код и `package.json`.

## Быстрый старт

Устанавливай зависимости отдельно в каждой части проекта:

```powershell
cd frontend
npm install

cd ..\backend
npm install
```

Запуск разработки:

```powershell
cd backend
npm run dev
```

```powershell
cd frontend
npm run dev
```

По умолчанию:

- backend API: `http://localhost:3000/api`
- frontend dev server: `http://localhost:5173`
- frontend axios base URL задан в `frontend/src/shared/api/axios.ts`
- backend CORS разрешает `http://localhost:5173`

## Проверки

Перед завершением задачи запускай проверки только для той части проекта, которую менял.

Frontend:

```powershell
cd frontend
npm run build
npm run lint
npm run format:check
```

Backend:

```powershell
cd backend
npm run type-check
npm run test
npm run lint
npm run format:check
```

Если проверку не удалось запустить из-за отсутствующих зависимостей, БД или переменных окружения, явно напиши это в финальном ответе.

## Архитектура frontend

- Используй алиас `@` для импортов из `frontend/src`.
- Основные слои:
  - `app/` - вход приложения, роуты, провайдеры, layouts, глобальные стили.
  - `pages/` - страницы.
  - `widgets/` - крупные блоки интерфейса.
  - `entities/` - бизнес-сущности, API, Redux slices, UI сущностей.
  - `shared/` - общие API-клиенты, хуки, типы, константы, UI.
- Для стилей компонентов используй CSS Modules рядом с компонентом.
- Не добавляй глобальные стили без необходимости.
- Для UI придерживайся существующего Ant Design/CSS Modules подхода.
- Для иконок используй `lucide-react`, если подходящая иконка уже есть.
- Не меняй маршрутизацию в `frontend/src/app/routes/appRoutes.tsx` без проверки protected/admin маршрутов.
- При изменении API-вызовов сверяйся с backend routers/controllers/services и `API_CONTRACTS.md`.

## Архитектура backend

- Сохраняй слойность: `model -> service -> controller -> router`.
- Основные папки:
  - `models/` - Sequelize модели.
  - `services/` - бизнес-логика.
  - `controllers/` - обработка request/response.
  - `routers/` - Express маршруты.
  - `mocks/` и `seeders/` - тестовые и seed данные.
  - `tests/integration/` - интеграционные тесты.
- Не размещай бизнес-логику в router.
- При добавлении endpoint обновляй router, controller, service и при необходимости `API_CONTRACTS.md`.
- Для валидации используй существующие утилиты в `src/utils/validation`.
- БД зависит от `.env`: `DB_NAME`, `DB_NAME_TEST`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`, `IS_TESTING_STATUS`.

## Правила изменений

- Сначала изучай существующий код и паттерны, потом меняй.
- Не переписывай модуль целиком, если задача локальная.
- Не удаляй и не откатывай изменения пользователя.
- Если в `git status` есть чужие изменения, работай вокруг них и упоминай конфликт только когда он мешает задаче.
- Не меняй `package-lock.json` без причины.
- Не добавляй новые библиотеки, если задачу можно решить существующими зависимостями.
- При изменении публичного API синхронизируй frontend, backend и документацию.
- Для пользовательских сценариев магазина проверяй роли: guest/user/admin/super-admin, если они затрагиваются.

## Тестирование и качество

- Для backend изменений в service/controller/router добавляй или обновляй Vitest/Supertest тесты, если поведение меняется.
- Для frontend изменений проверяй TypeScript build; для важных пользовательских потоков добавляй Playwright тесты или ручную проверку через dev server.
- При исправлении багов добавляй регрессионную проверку там, где это разумно.
- Не оставляй временные `console.log`, тестовые данные или закомментированные хаки.

## Как агенту отчитываться

В финальном ответе указывай:

- что изменено;
- какие проверки запущены;
- что не удалось проверить и почему;
- какие файлы важно посмотреть дальше, если есть продолжение.

## Что можно улучшать в этом файле

Добавляй сюда правила, которые повторяются в работе:

- команды запуска Docker/БД, если они появятся;
- точный формат `.env.example`;
- правила именования компонентов, slices, services и тестов;
- список критичных пользовательских сценариев для ручной проверки;
- договоренности по API response shape;
- чеклист релиза или деплоя;
- правила работы с Yandex Auth;
- правила для админки и ролей;
- частые ошибки проекта и способы их диагностировать.

Хорошее правило для `AGENTS.md` должно быть конкретным: команда, путь, ограничение, чеклист или пример. Не добавляй общие советы, которые агент не сможет применить.
