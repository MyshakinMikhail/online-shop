План изучения и внедрения в интернет-магазин
Стадия 0. Подготовка к собесу — 2–3 часа

Цель: уметь нормально рассказать, зачем это нужно.

Выучи формулировку:

Я хочу добавить observability в магазин: логирование ошибок, сбор пользовательских событий, метрики производительности и мониторинг API. Это нужно, чтобы понимать, что происходит с приложением после выката: где пользователи ломаются, какие страницы тормозят, какие запросы падают и какие изменения реально улучшают продукт.

Что знать:

логи отвечают: что произошло;
метрики отвечают: насколько всё хорошо/плохо;
трейсинг отвечает: где именно в цепочке тормозит запрос;
алерты отвечают: когда надо срочно реагировать.

Стадия 1. Деплой магазина — 3–5 часов
Что сделать

Frontend:

залить проект на GitHub;
настроить build;
задеплоить на Vercel;
получить публичный URL.

Backend:

если backend на Node/Express — Render;
если есть PostgreSQL — Render Postgres/free alternative;
вынести .env;
настроить CORS между frontend URL и backend URL.
Что уметь сказать на собесе

Сейчас проект локальный, но я понимаю, что для логов и метрик лучше иметь хотя бы staging/prod окружение. Поэтому я бы начал с бесплатного деплоя: frontend на Vercel, backend на Render, env-переменные отдельно, CORS настроен под домен фронта.

Стадия 2. Логирование ошибок через Sentry — 2–4 часа
Что добавить

На frontend:

подключить Sentry;
ловить runtime errors;
ловить unhandled promise rejection;
передавать environment: local, staging, production;
передавать release/version.

Примеры ошибок:

упала страница товара;
ошибка при добавлении в корзину;
API вернул 500;
сломался checkout.
Что это даст

Ты сможешь видеть:

stack trace;
страницу, где произошла ошибка;
браузер;
пользователя;
версию приложения.
Что сказать на собесе

Я бы начал с Sentry, потому что это быстрый способ увидеть реальные frontend-ошибки в production. Особенно важно логировать checkout, корзину и авторизацию, потому что там ошибки напрямую влияют на бизнес.

Стадия 3. Пользовательские события — 3–5 часов

Это уже ближе к продуктовой аналитике.

Что логировать в магазине
product_viewed;
product_added_to_cart;
cart_opened;
checkout_started;
order_created;
favorite_added;
search_used;
filter_changed.

Пример структуры события:

{
  event: "product_added_to_cart",
  productId: "123",
  categoryId: "shoes",
  price: 4990,
  userId: "42",
  timestamp: Date.now()
}
Зачем

Ты сможешь понять:

какие товары смотрят;
где пользователи отваливаются;
сколько людей дошли до заказа;
какие фильтры реально используют.
Что сказать на собесе

Для интернет-магазина я бы отдельно логировал воронку: просмотр товара → добавление в корзину → начало checkout → успешный заказ. Это помогает не просто чинить ошибки, а понимать поведение пользователей.

Стадия 4. Метрики производительности — 2–4 часа
Что изучить

Core Web Vitals:

LCP — как быстро появился главный контент;
CLS — прыгает ли интерфейс;
INP — насколько быстро интерфейс реагирует.
Что добавить
измерение Web Vitals;
логирование медленных страниц;
отслеживание времени загрузки каталога;
замер времени ответа API.
Что сказать

Для магазина важна скорость: если каталог или страница товара долго грузится, пользователь может уйти. Поэтому я бы собирал LCP, INP, CLS и отдельно замерял время загрузки товаров.

Стадия 5. Backend-логи и API-метрики — 4–6 часов
Что добавить на backend

Логи:

входящий запрос;
метод;
URL;
status code;
duration;
userId, если есть;
requestId.

Пример:

GET /api/products 200 143ms userId=42 requestId=abc-123

Метрики:

количество запросов;
количество ошибок 4xx/5xx;
среднее время ответа;
самые медленные endpoint’ы.
Что сказать

На backend я бы добавил request logging и requestId. Тогда можно связать ошибку на фронте с конкретным запросом на backend.

Это очень сильная мысль для собеса.

Стадия 6. Feature flags — 3–5 часов

Это особенно близко к команде A/B-тестирования.

Что сделать

Например, у тебя есть новая карточка товара.

Вместо того чтобы сразу включать всем:

if (featureFlags.newProductCard) {
  return <NewProductCard />;
}

return <OldProductCard />;
Зачем
можно включить фичу только себе;
можно включить 10% пользователей;
можно быстро выключить без деплоя;
можно проводить A/B-тест.
Что сказать

Feature flags нужны, чтобы безопасно выкатывать изменения. Особенно это важно для A/B-тестов: можно показывать разным группам пользователей разные варианты интерфейса и сравнивать метрики.

Стадия 7. Мини A/B-тест — 4–8 часов
Что сделать в магазине

Например:

старая кнопка “Купить”;
новая кнопка “В корзину”.

Пользователя делишь на группу A или B:

const group = userIdHash % 2 === 0 ? "A" : "B";

И логируешь:

{
  event: "checkout_started",
  abTest: "buy_button_text",
  group: "B"
}
Что измерять
добавление в корзину;
начало оформления;
успешный заказ.
Что сказать

Я бы сделал простой A/B-тест на карточке товара: разные тексты или расположение кнопки, а потом сравнил конверсию в добавление в корзину. Важно, чтобы пользователь стабильно попадал в одну и ту же группу, например через hash от userId.

Приоритет на ближайшие 27 часов

До собеса не надо всё реализовывать. Тебе нужно уметь умно рассказать.

Я бы сделал так:

2 часа	Повторить Sentry + frontend error logging
2 часа	Повторить Core Web Vitals
2 часа	Разобрать feature flags и A/B-тесты
2 часа	Продумать, как это встроить в твой магазин

Главная фраза на завтра:

Я понимаю, что frontend — это не только сверстать интерфейс. Особенно во внутренних платформах важно уметь наблюдать за системой: логировать ошибки, собирать метрики, понимать пользовательские сценарии, безопасно выкатывать изменения через feature flags и проверять гипотезы через A/B-тесты. Я как раз хотел добавить это в свой интернет-магазин.

Это звучит очень близко к их команде.
